Codex pairing med marimo

Pairing er det som gjorde at Codex kunne arbeide inne i den aktive notebooken, ikke bare redigere Python-filen på disk. Det er nyttig når vi vil at agenten skal se notebook-state, kjøre celler, eksportere tabeller eller gjøre raske eksperimenter direkte i marimo.

marimo browser Notebooken kjører på localhost.
Pair-dialog Velg Codex og kopier kommando/token.
marimo-pair skill Installeres med npx skills add.
Codex Agenten sender kode inn i notebook-kernelen.
Live output Endringer og resultater vises i browseren.
Installer pair-skillen
npx skills add marimo-team/marimo-pair
Kommandoen marimo viser i Pair with an agent
codex "$(uvx marimo@latest pair prompt \
  --url 'http://localhost:2718/' \
  --with-token \
  --codex)"
Token: marimo viser en kort token i Pair-dialogen. Den skal ikke committes. Den brukes bare for å koble agenten til den lokale, kjørende notebook-sessionen.
Hva agenten kan gjøre når pairing virker
# Eksempler på oppgaver Codex kan gjøre i en paired notebook:
# - kjøre alle celler på nytt
# - lese variabler som top_customers eller segment_summary
# - legge til nye celler
# - eksportere tabeller til Excel
# - teste DuckDB/Parquet-kode i browser-runtime
# - sende mo.status.toast(...) som synlig bekreftelse
Praktisk regel: Når notebooken kjører og er paired, bør celler endres gjennom marimo-kernelen. Direkte filskriving kan bli overskrevet av marimo dersom den levende notebook-state er nyere enn filen.

Azure Static Web Apps med Parquet-snapshots

GitHub Pages-sporet over fungerer fint for åpne, statiske demoer. Hvis vi allerede har Azure Static Web Apps tilgjengelig, kan vi bruke samme marimo/WASM-prinsipp der: appen er statisk, mens Postgres bare brukes i build-steget for å lage Parquet-snapshotet.

Azure Postgres Database/view med read-only tilgang for snapshot-jobben.
GitHub Actions Henter secret og bygger datafilen ved push eller schedule.
Polars Leser Postgres og skriver komprimert Parquet.
Azure SWA Publiserer marimo HTML/WASM-assets og data/*.parquet.
Browser DuckDB-WASM kjører SQL lokalt mot snapshotet.
Hvorfor SWA

God Azure-integrasjon

SWA passer godt hvis appen skal leve i Azure-landskapet vårt, bruke egne domener, tilgangsstyring eller senere få et server-side API.

Hvorfor snapshot

Ingen live DB-avhengighet

Sluttbrukeren belaster ikke Postgres. Appen laster bare statiske filer fra SWA, og all filtrering/spørring skjer i browseren.

Hvorfor DuckDB

Raske lokale queries

Parquet holder data kompakt, og DuckDB-WASM gir SQL over snapshotet uten backend-kall for hvert filter eller dashboardvalg.

Hovedregel: selv på Azure SWA bør marimo-appen ikke koble direkte til Postgres fra nettleseren. Connection string ligger i GitHub/Azure secrets og brukes bare av build-jobben.

SWA workflow for snapshot-modellen

SWA-deployen ligner GitHub Pages, men bruker Azure sin deploy-action og en deployment token. Først bygger vi data til dist/data, deretter eksporterer vi marimo-appen til dist/index.html, og til slutt laster vi hele dist/ opp til SWA.

.github/workflows/deploy-azure-swa.yml
name: Deploy marimo app to Azure Static Web Apps

on:
  push:
    branches: [main]
  workflow_dispatch:
  schedule:
    - cron: "15 5 * * *"  # daglig snapshot, juster ved behov

jobs:
  build_and_deploy:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4

      - uses: astral-sh/setup-uv@v6

      - name: Build Postgres snapshot
        env:
          POSTGRES_URL: ${{ secrets.POSTGRES_URL }}
        run: |
          rm -rf dist
          mkdir -p dist/data
          uv run --with polars --with connectorx --with pyarrow \
            python scripts/export_snapshot.py

      - name: Export marimo WASM app
        run: |
          uv run marimo export html-wasm customer_churn_demo.py \
            -o dist/index.html \
            --mode run \
            --no-show-code \
            --force
          rm -f dist/CLAUDE.md dist/AGENTS.md dist/agents.md

      - name: Deploy to Azure Static Web Apps
        uses: Azure/static-web-apps-deploy@v1
        with:
          azure_static_web_apps_api_token: ${{ secrets.AZURE_STATIC_WEB_APPS_API_TOKEN }}
          repo_token: ${{ secrets.GITHUB_TOKEN }}
          action: upload
          app_location: dist
          output_location: ""
          skip_app_build: true
Secrets: POSTGRES_URL brukes av snapshot-scriptet. AZURE_STATIC_WEB_APPS_API_TOKEN hentes fra Azure SWA-ressursen og brukes bare av deploy-steget.
Oppdatering: data oppdateres ved push, manuell workflow_dispatch, eller schedule. Bruk schedule hvis appen skal ha faste dagsferske snapshots uten kodeendringer.

GitHub Pages vs Azure SWA

Valg Bruk når Datamønster
GitHub Pages Demoer, åpne interne prototyper, enkel publisering fra repo. Samme Parquet-snapshot. Ingen secrets i browser.
Azure Static Web Apps Azure-nær drift, egne domener, tilgangsstyring, mulig senere API. Samme Parquet-snapshot. Build-jobb henter fra Azure Postgres.
SWA + live API Når data må være live og brukervalg skal treffe server-side queries. Ikke snapshot. Krever Azure Functions/API og mer drift.
Anbefaling nå: bruk Azure SWA med Parquet-snapshot for apper der data kan være oppdatert ved deploy eller på schedule. Det gir lav kompleksitet og god ytelse uten live databaseeksponering.

Hva vi har validert

Spørsmål Resultat Konsekvens
Kan marimo eksporteres til en installasjonsfri app? Ja, med marimo export html-wasm. Mottaker trenger bare nettleser, men filene må serveres over HTTP.
Kan GitHub Pages brukes? Ja, med GitHub Actions deploy. Push til main kan automatisk bygge og publisere appen.
Kan appen hente direkte fra Postgres? Ikke anbefalt i en statisk browser-app. Credentials ville blitt offentlige, og Postgres er ikke et browser-API.
Kan DuckDB kjøre i marimo WASM? Ja, testet i browser med SQL og tabelloutput. Vi kan gjøre lokale SQL-spørringer i appen.
Kan DuckDB lese statisk Parquet? Ja, når Parquet først fetches til Pyodide-filsystemet. Bruk pyfetch("../data/file.parquet") og skriv til /tmp.

Polars-basert dataeksport fra Postgres

I build-steget bør vi hente data fra Postgres og skrive en optimalisert Parquet-fil. Eksempelet under bruker Polars, og kan kjøres i GitHub Actions.

Eksempel: scripts/export_snapshot.py

Scriptet leser connection string fra miljøvariabel, henter et begrenset view, gjør enkle transformasjoner og skriver Parquet til dist/data/.

Prinsipp: Dataene som publiseres kan være offentlige. Secret-en skal aldri inn i notebooken eller i dist/.
scripts/export_snapshot.py
from pathlib import Path
import os
import polars as pl

POSTGRES_URL = os.environ["POSTGRES_URL"]
OUT = Path("dist/data/customer_snapshot.parquet")
OUT.parent.mkdir(parents=True, exist_ok=True)

query = """
select
  customer_id,
  segment,
  monthly_revenue,
  support_cases,
  tenure_months,
  usage_score,
  churn_risk
from analytics.customer_churn_snapshot
where is_active = true
"""

df = pl.read_database_uri(query=query, uri=POSTGRES_URL)

snapshot = (
    df
    .with_columns(
        expected_loss=(pl.col("monthly_revenue") * pl.col("churn_risk")).round(0),
        risk_bucket=pl.when(pl.col("churn_risk") < 0.12).then(pl.lit("Lav"))
          .when(pl.col("churn_risk") < 0.25).then(pl.lit("Middels"))
          .otherwise(pl.lit("Hoy")),
    )
    .select(
        "customer_id",
        "segment",
        "monthly_revenue",
        "support_cases",
        "tenure_months",
        "usage_score",
        "churn_risk",
        "expected_loss",
        "risk_bucket",
    )
)

snapshot.write_parquet(
    OUT,
    compression="zstd",
    statistics=True,
)

print(f"Wrote {snapshot.height:,} rows to {OUT}")

GitHub Actions: bygg data og app

Workflowen må gjøre to ting: lage Parquet-snapshotet først, og deretter eksportere marimo-appen. Appen og datafilen lastes opp som ett Pages-artifact.

.github/workflows/deploy-pages.yml
name: Deploy marimo app to GitHub Pages

on:
  push:
    branches: [main]
  workflow_dispatch:

permissions:
  contents: read
  pages: write
  id-token: write

jobs:
  build:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4

      - uses: astral-sh/setup-uv@v6

      - name: Build data snapshot from Postgres
        env:
          POSTGRES_URL: ${{ secrets.POSTGRES_URL }}
        run: |
          mkdir -p dist/data
          uv run --with polars --with connectorx --with pyarrow \
            python scripts/export_snapshot.py

      - name: Export marimo WASM app
        run: |
          uv run marimo export html-wasm customer_churn_demo.py \
            -o dist/index.html \
            --mode run \
            --no-show-code \
            --force
          rm -f dist/CLAUDE.md dist/AGENTS.md dist/agents.md

      - uses: actions/configure-pages@v5

      - uses: actions/upload-pages-artifact@v3
        with:
          path: dist

  deploy:
    needs: build
    runs-on: ubuntu-latest
    environment:
      name: github-pages
      url: ${{ steps.deployment.outputs.page_url }}
    steps:
      - id: deployment
        uses: actions/deploy-pages@v4
Merk: Dette gir snapshots. Hvis data må være live, trenger vi en backend/API mellom appen og Postgres.

marimo-app med DuckDB-WASM og Parquet

I browseren må Parquet-filen hentes som bytes og skrives til Pyodide sitt lokale filsystem før DuckDB kan lese den. I GitHub Pages ligger appens worker under assets/, derfor bruker vi ../data/....

customer_churn_demo.py
# /// script
# requires-python = ">=3.11"
# dependencies = [
#   "duckdb>=1.2.0",
#   "marimo>=0.23.3",
#   "polars>=1.0.0",
# ]
# ///

import marimo

app = marimo.App(width="full")


@app.cell
def _():
    import duckdb
    import marimo as mo
    import polars as pl
    from pathlib import Path
    from pyodide.http import pyfetch
    return Path, duckdb, mo, pl, pyfetch


@app.cell
async def _(Path, pyfetch):
    response = await pyfetch("../data/customer_snapshot.parquet")
    parquet_path = Path("/tmp/customer_snapshot.parquet")
    parquet_path.write_bytes(await response.bytes())
    return parquet_path


@app.cell
def _(duckdb, parquet_path, pl):
    segment_summary = duckdb.sql(f"""
        select
          segment,
          count(*) as customers,
          sum(monthly_revenue) as revenue,
          avg(churn_risk) as avg_risk,
          sum(expected_loss) as expected_loss
        from read_parquet('{parquet_path}')
        group by segment
        order by expected_loss desc
    """).pl()

    # DuckDB returns an Arrow-backed result; keep working in Polars.
    segment_summary = segment_summary.with_columns(
        pl.col("avg_risk").round(3),
        pl.col("revenue").round(0),
        pl.col("expected_loss").round(0),
    )
    return segment_summary


@app.cell
def _(mo, segment_summary):
    mo.ui.table(segment_summary, page_size=20)
    return

Figur 3: hvor koden kjører

Det viktigste skillet er build-time versus runtime. Secrets og Postgres-tilgang finnes bare i build-time. Sluttbrukeren får bare ferdig app og ferdige data.

Build-time GitHub Actions, secrets, Postgres, Polars, Parquet-skriving.
Static hosting GitHub Pages serverer HTML, JS, WASM-assets og Parquet.
Runtime Brukerens nettleser kjører marimo, Pyodide og DuckDB-WASM.

Beslutningsregler

1

Bruk statisk Parquet når data kan være et snapshot

Dette er enklest, raskest å hoste og tryggest for database-credentials.

2

Bruk backend/API når data må være live

GitHub Pages er statisk. Live database krever et serverledd med auth og rate limiting.

3

Ikke legg connection string i notebooken

Selv read-only credentials blir offentlige hvis de legges i en WASM/Pages-app.

4

Bruk width="full" for tabell- og dashboard-apper

Det gir bedre arbeidsflate og reduserer unødvendig horisontal scrolling.
